# Educational Content Integration — Architecture Spec

> Companion to PLAN.md / CORPUS.md / REVEAL.md. This doc covers the *real-world, cited, educational* content layer — the inoculation payload — and how it threads through the UI via ticker, tooltips, codex, and post-action notifications.

**Critical distinction:** `CORPUS.md` already specifies the *in-fiction* content (fake headlines the player generates). This doc covers the *out-of-fiction* content (real, sourced facts about real disinformation history and research). They are separate systems with separate files.

**Cross-reference mechanism (optional but encouraged).** A templated fake headline in CORPUS.md can optionally carry an `inspiredBy: <codex-id>` field. When that headline is rendered in-game, a small "ⓘ" glyph appears in the corner of the post card; clicking opens the linked codex entry. This is the bridge between "the technique you just used" and "here's the documented real-world case." Not every fake headline needs one — but flagship pieces (the wedge-issue rally, the deepfake whistleblower, the wellness anti-vax thread) should.

## 1. Content Directory Structure

```
content/
├── facts/                       # ticker corpus — one-liners
│   ├── research.json            # MIT/Cambridge/Stanford findings
│   ├── platform-mechanics.json  # algorithm research
│   └── historical.json          # Cold War, Nazi propaganda research
├── codex/                       # long-form entries (2–4 paragraphs)
│   ├── operation-denver.md
│   ├── internet-research-agency.md
│   ├── disinformation-dozen.md
│   ├── cambridge-analytica.md
│   ├── pizzagate-qanon.md
│   ├── macedonian-content-farms.md
│   ├── meaningful-social-interactions.md
│   ├── tiktok-rabbit-hole.md
│   ├── youtube-radicalization.md
│   └── ... (one .md per entry)
├── tooltips/                    # short "Based on:" lines for upgrades/assets
│   └── precedents.json          # keyed by upgrade/asset id
├── triggers/                    # action-id → codex-id mapping
│   └── postaction.json
└── index.json                   # generated at build: id → metadata for all of the above
```

### 1.1 File format choices (and why)

| Layer    | Format | Why                                                                           |
|----------|--------|-------------------------------------------------------------------------------|
| Facts    | JSON   | Tiny records, type-checked, easy to filter/shuffle. No prose styling.        |
| Codex    | MD+frontmatter | Long prose authoring is miserable in JSON. Frontmatter holds metadata. |
| Tooltips | JSON   | Many short records keyed by upgrade id. No prose styling.                    |
| Triggers | JSON   | Pure lookup table.                                                            |
| Index    | JSON   | Generated by a Vite plugin at build; ships ~2–5 KB.                          |

### 1.2 Canonical schemas

**`facts/*.json`** — array of:
```ts
{
  id: string;              // stable, kebab-case, globally unique
  text: string;            // single sentence
  source: string;          // "MIT, Vosoughi et al. 2018"
  sourceUrl?: string;      // optional canonical link
  tags: string[];          // ["research", "platform"], plus political balance tag
  era?: PhaseId;           // gates when this fact first appears in ticker
  technique?: DepictId;    // optional DEPICT mapping for post-game debrief
}
```

**`codex/*.md`** — frontmatter + body:
```yaml
---
id: operation-denver
title: "Operation Denver (KGB, 1980s)"
era: cable               # earliest era this entry unlocks in
unlockTriggers:
  - upgrade: conspiracy-tier-5
  - upgrade: impersonation-tier-4
techniques: [conspiracy, impersonation]
politicalFlavor: foreign-troll
sources:
  - title: "Operation Denver: KGB's AIDS Disinformation Campaign"
    publisher: "U.S. State Department"
    year: 1987
  - title: "Active Measures Working Group Report"
    publisher: "U.S. Government"
    year: 1992
related: [internet-research-agency, plandemic-2020]
estimatedReadTime: 90   # seconds
---

(2–4 paragraphs of body text — see §7 style guide)
```

**`tooltips/precedents.json`** — keyed by entity id:
```ts
{
  "asset.botnet.tier3": {
    text: "In 2016, the Russian IRA employed ~80 full-time staff posing as Americans, with a budget of roughly $1.25M/month.",
    source: "Senate Intel Cmte Vol. II, 2019",
    codexEntry: "internet-research-agency"   // click-through target
  },
  ...
}
```

**`triggers/postaction.json`**:
```ts
{
  "first-buy:botnet.tier3": "internet-research-agency",
  "first-synergy:E+P": "meaningful-social-interactions",
  "first-buy:deepfake-generator": "ai-deepfakes-2024",
  ...
}
```

## 2. `Ticker.svelte` — Top-bar scrolling fact ticker

### 2.1 Behavior

- One fact at a time, horizontal scroll (right-to-left for LTR; CSS `transform: translateX`).
- Pace: 1 fact per ~25 seconds. Scroll speed = `(text-width + viewport-width) / 25s`.
- Picker draws from `facts/` files filtered by `era <= currentEra`, balanced via the **dimensional** inverse-frequency weighting (per §7.4).
- **Repetition guard:** maintain a `recentlySeen` ring buffer of last 20 fact ids; reject draws that hit the buffer.
- **Click-to-pause and click-to-expand.** Click anywhere on the ticker freezes the current fact and slides up a 2-line expansion: source + "Read more →" link to the matching codex entry. Click again or auto-resume after 8 seconds of no interaction.
- **Highlighted facts.** A subset of facts carry `highlighted: true` in their JSON record. These get a subtle visual treatment — slower scroll (1.5× duration), one-shade-brighter background, faint left-edge accent stripe — so they don't drift past unread. Use this sparingly (~15% of pool) and prefer cross-dimensional facts that puncture lazy assumptions.
- Pauses on hover (desktop) or tap (mobile).
- Respects `prefers-reduced-motion: reduce`: snap-to-position transitions every 25s instead of continuous scroll.

(These two affordances — click-to-pause and highlighted facts — exist because in v0.1 the ticker is doing 100% of the proactive educational lifting; tooltips are passive. Without them the ticker risks becoming background noise.)

### 2.2 Component contract

```ts
// Ticker.svelte props
{
  pace?: number;            // seconds per fact, default 25
  filter?: (f: Fact) => boolean;  // optional; defaults to era-gated
}
```

### 2.3 Visual treatment

- 24px tall strip directly under the resource topbar.
- Background: 1 shade darker than topbar; not eye-catching.
- Text: 12px, tabular-numeral monospace pairs nicely with the rest of the UI.
- Source citation rendered inline in muted color: `Source: MIT, 2018`.
- Subtle right-edge fade-out gradient to disguise the scroll boundary.

### 2.4 Lifecycle

- Mounted once at app start, lives for the session.
- Subscribed to `currentEra` reactive — when era changes, the eligible pool grows (does not shrink).
- On reveal trigger (Mebro), the ticker fact pool **rotates to a reveal-only set** — facts about inoculation research, fact-checking, lateral reading. The ticker becomes part of the reveal.

## 3. `Codex.svelte` — Side panel with unlock-gated entries

### 3.1 Behavior

- Collapsible panel docked to the right (desktop) or accessed via tab nav (mobile).
- Two states: collapsed (icon only — small book glyph with a numeric badge for unread) and expanded (overlay/sheet).
- Entry list grouped by era; entries before they're unlocked render as `???` with the unlock hint ("Buy your first botnet to unlock") — matches PLAN.md §6f gradual-reveal pattern.
- Reading an entry sets `read: true` flag (persisted), suppresses the unread badge.
- Search box (entry title + tags). No full-text search in MVP — overkill for ~150 entries.
- Each entry shows: title, era badge, technique tags, estimated read time, body, sources block, "Related entries" links.

### 3.2 Unlock logic

```ts
// content/codex.ts (engine module)
type UnlockTrigger =
  | { type: 'upgrade'; id: string; minLevel?: number }
  | { type: 'asset'; id: string; minCount?: number }
  | { type: 'phase'; id: PhaseId }
  | { type: 'synergy'; pair: [DepictId, DepictId] }
  | { type: 'cure'; min: number };

function isUnlocked(entry: CodexEntry, state: GameState): boolean {
  return entry.unlockTriggers.some(t => satisfies(t, state));
}
```

- Re-evaluated each tick (cheap — small set, simple checks). Newly unlocked entries:
  1. Append to a `newlyUnlocked` queue.
  2. Drained by the Notification system (§5) which surfaces them.
  3. Badge increments on the Codex icon.

### 3.3 Markdown rendering

- Use a tiny MD renderer (e.g. `marked` at ~50KB minified — but evaluate against budget; PLAN.md targets < 100KB initial JS).
- **Alternative recommended:** ship codex bodies as pre-compiled HTML at build time via a Vite plugin (`vite-plugin-md` or a 30-line custom transformer). Zero runtime MD-parsing cost.
- Sanitize on build (DOMPurify at build time, not runtime). All codex MD is first-party authored, so risk is low — but build-time sanitization catches our own mistakes.

### 3.4 Reveal-state behavior

When the Mebro reveal fires, the Codex gains:
- A new top section: **"Counter-techniques"** — 6 entries (one per DEPICT) auto-unlocked, each describing the real-world inoculation/fact-checking practice that defeats it.
- The codex icon pulses gently to draw attention.
- The debrief screen (REVEAL.md §6) deep-links into the relevant counter-technique codex entries.

## 4. Tooltip Extension Spec — `Tooltip.svelte` with "Based on:"

### 4.1 Current state (from PLAN.md / REVEAL.md)

The existing UpgradeCard / PlatformCard already have tooltips for mechanical effects. We extend.

### 4.2 Extension

Tooltips gain a second section below the mechanical effect block:

```
┌────────────────────────────────────────────────┐
│ Foreign Troll Farm                             │
│ +10% Followers globally, ×1.5 heat             │
│ Cost: 500 Fol  Owned: 0 → 1                    │
├────────────────────────────────────────────────┤
│ Based on:                                      │
│ In 2016, the Russian IRA employed ~80 staff    │
│ posing as Americans, ~$1.25M/month budget.     │
│ — Senate Intel Cmte Vol. II, 2019              │
│ [Read full entry →]                            │
└────────────────────────────────────────────────┘
```

### 4.3 Loading

- Tooltip precedent lookup is synchronous from `tooltips/precedents.json` (pre-loaded — ~10 KB total).
- "Read full entry" link triggers a Codex panel open with the relevant entry pre-selected.

### 4.4 Mobile

- Long-press (400ms) opens the tooltip as a bottom-sheet modal.
- Tap-anywhere-else dismisses.
- Long-press must not interfere with the bulk-buy long-press shortcut (§12 in PLAN.md) — bulk-buy long-press fires on *buttons*, tooltip long-press fires on *card body*. Disambiguate via target.

### 4.5 Coverage

Not every upgrade has a precedent (some are purely mechanical). When `tooltips/precedents.json[id]` is missing, the "Based on:" section is omitted. Acceptable. Aim for ~60% coverage on assets, ~40% on DEPICT upgrades by ship.

## 5. `Notification.svelte` — Post-action educational popups

### 5.1 Trigger taxonomy

| Trigger key                    | Fires when                                              | Codex entry         |
|--------------------------------|----------------------------------------------------------|---------------------|
| `first-buy:<id>`              | Player buys an asset/upgrade for the first time          | from triggers JSON  |
| `first-synergy:<E+P>`         | A DEPICT synergy activates for the first time            | combo-themed entry  |
| `first-platform:<id>`         | A new platform unlocks                                   | platform-mechanics  |
| `first-event:<id>`            | A random event of a specific type fires                  | technique-themed    |
| `cure-threshold:<n>`          | Cure crosses 0.20 / 0.40 / 0.60 / 0.80                   | inoculation-research|
| `phase-transition:<id>`       | New era begins                                           | era-overview        |

### 5.2 UI behavior

- Toast-style, top-right (or top-center on mobile).
- ~6 second auto-dismiss; pauses on hover.
- Body: 1 line plain-language hook + small "Click to learn more" link.
- Click expands to the Codex entry; never modal-blocks gameplay.
- Stack max 3 simultaneous; surplus queued. Queue drains at ~1 every 8 seconds to avoid overwhelm.

### 5.3 Frequency cap

Players who power through Phase 1–3 in 15 minutes shouldn't see 40 notifications. **Cap: at most 1 educational notification per 90 seconds of wall-clock time.** Excess get marked "deferred" — surfaced as a quiet "3 new codex entries unlocked" badge on the Codex icon, no toast. The codex itself shows them as newly-unread.

### 5.4 Anti-pattern guardrails

- **Never** during the Mebro reveal sequence (first 60s of reveal). The reveal is its own piece of UI choreography (REVEAL.md §2) — competing toasts would dilute it.
- **Never** stack on top of random-event banners — the random event already occupies the same screen real estate. Defer.
- **Never** during the endgame debrief screen — the debrief IS the educational payload at that point.

## 6. Content Loading Strategy

### 6.1 Recommendation: **static JSON per entry + manifest, fetched on demand**

(Earlier draft proposed a custom Vite plugin. Walked back — overengineered. Plain static files do the job with far less moving machinery.)

Three tiers:

| Tier               | What                                          | When loaded         | Size estimate     |
|--------------------|-----------------------------------------------|---------------------|-------------------|
| **A — inline**     | Ticker facts, tooltips JSON                   | Initial bundle      | ~15 KB gzipped    |
| **B — manifest**   | `/public/codex/manifest.json` (id, title, era, tags, unlock condition, ~100-char summary per entry) | App boot via fetch  | ~10–20 KB gzipped |
| **C — entry body** | `/public/codex/<id>.json` (one file per entry, full body + frontmatter) | On open in Codex panel | ~5–15 KB per entry |
| **D — external**   | Source URLs                                   | On user click       | 0 (external tab)  |

### 6.2 Why this split

- Codex bodies live as **static JSON files in `/public/codex/`** — Vite copies them as-is, GH Pages serves them, no plugin needed. Each is ~5–15 KB; 300 entries totals 1.5–4.5 MB *on disk*, but the player only fetches the ones they open.
- The **manifest** is the single boot-time fetch (~15 KB). It powers the codex index, unlock predicates, and the post-action notification targeting. Manifest IS the "index" — no separate index file.
- **Entry bodies** are MD pre-rendered to HTML strings inside the JSON at authoring time (a build script does `marked + DOMPurify` once, writes the output). Runtime gets clean HTML strings, no MD parser shipped.
- Caching: HTTP-level (the JSON files are static, immutable URLs once filenames include a content hash) plus in-memory after first fetch.

### 6.3 Build-time CI script (not a Vite plugin)

A small Node script (~80 lines, runs in CI + as `npm run content:build`):

1. Walks `content/codex/*.md`.
2. Parses frontmatter (gray-matter, dev-dep only).
3. Renders body MD → HTML (marked + DOMPurify, dev-deps only).
4. Validates: every required frontmatter field present, citations have at least a title + publisher, `related` entries resolve, `unlockTriggers` reference real game ids.
5. Runs the dimensional-coverage check (§7.5).
6. Writes `/public/codex/<id>.json` for each entry.
7. Writes `/public/codex/manifest.json` (metadata-only summary).

CI fails on any validation error. No runtime Vite plugin to maintain — just static files + a Node build step.

### 6.4 Tradeoff considered

| Option                          | Pros                                       | Cons                                            |
|---------------------------------|--------------------------------------------|--------------------------------------------------|
| Full pre-bundle                 | Zero open latency, simpler runtime         | Blows 100 KB initial budget                      |
| Custom Vite plugin + JS chunks  | Code-split per entry, automatic            | Plugin code to maintain; runtime import() ceremony|
| **Static JSON + manifest + Node build script** | Minimal runtime, simple fetch, easy debug | Bodies must be valid JSON-with-HTML strings (build script handles) |

**Verdict: static JSON.** Same network shape as code-splitting, dramatically simpler engine.

## 7. Style Guide for Writing Fact Entries

### 7.1 Voice

- **Clinical, not breathless.** "X happened in Y." Not "X shockingly happened in Y."
- **Past tense for events**, present tense for ongoing mechanisms.
- **Names get full credit**: people, organizations, research institutions are named. No "a researcher" or "a study."
- **Numbers preferred over adjectives**: "11.2 million engagements" not "viral engagement."

### 7.2 Length

| Surface      | Target            | Hard ceiling          |
|--------------|-------------------|------------------------|
| Ticker fact  | 1 sentence        | 180 chars              |
| Tooltip      | 1–2 sentences     | 280 chars              |
| Codex entry  | 2–4 paragraphs    | ~600 words             |

If a codex entry runs longer, split it. Codex entries cross-link cleanly.

### 7.3 Sourcing standard

- **Every fact requires at least one citation.** No exceptions, even for "common knowledge" items.
- Citation hierarchy (use highest available):
  1. Peer-reviewed academic paper (preferred — Science, Nature, PNAS, JCMC, etc.)
  2. Government report (Senate Intelligence Committee, FBI, EU DisinfoLab, etc.)
  3. Established research institute (Stanford Internet Observatory, Cambridge SDML, DFRLab, Graphika, Mozilla Foundation, RAND)
  4. Major newspaper of record (NYT, AP, Reuters, WaPo, BBC, Guardian)
  5. Specialist publication (NiemanLab, Columbia Journalism Review, Wired, MIT Tech Review)
- Citation format inline: `— Vosoughi et al., Science vol. 359, 2018`
- A `sourceUrl` field is *optional* but encouraged. Use canonical URLs (DOI, gov publication PDF) not aggregators.
- **If sources conflict, say so.** Cambridge Analytica's psychographic efficacy is the canonical example — claim and counter-claim go in the same paragraph.

### 7.4 Political balance — enforcement (dimensional, NOT partisan-quota)

(Walked back from an earlier draft that used partisan quotas. Quotas force false-equivalence: a lot of the strongest case material — Operation Denver, the Disinformation Dozen, Macedonian content farms, QAnon's blood-libel roots — doesn't sit on a left/right axis at all. And empirical asymmetry in documented operations is real; a forced 50/50 partisan corpus is its own distortion. The fix is to require *dimensional* diversity, not partisan parity.)

**Required dimensional coverage.** Every entry tags one or more `dimensions` from:

```
foreign-state         # Russia, China, Iran, etc.
domestic-political    # any direction; tag the specific actor, not the side
commercial-grift      # wellness, supplements, ad-revenue farms
historical-pre-internet  # Soviet active measures, Nazi propaganda research, COINTELPRO
platform-mechanic     # algorithm behavior, recommender pathology
ai-generated          # deepfakes, generative-AI content ops
```

Build-time checks (CI gate):

1. **Dimensional floor.** At least 5 of 6 dimensions present, each with ≥ 8% of entries.
2. **Source diversity.** No single outlet provides > 30% of citations. No single political-leaning bucket (per AllSides or Ad Fontes classification) provides > 50% of citations.
3. **Per-DEPICT-technique dimensional coverage.** Each DEPICT technique has examples from ≥ 3 dimensions. (Polarization is the critical case — the "both sides do it" point only lands when both are actually documented.)
4. **Tone scan.** Flag entries whose ratio of editorial adjectives to citations exceeds a threshold (linter regex against a short denylist: "shockingly", "notorious", "debunked" outside quoted source material, etc.). Flag, don't fail — author reviews.

**No editorializing.** "Wakefield's paper was retracted in 2010 after the BMJ documented fabricated data" is fine. "Wakefield's discredited paper" is editorializing — the reader already infers the verdict from the facts.

### 7.5 What the dimensional approach is and isn't

- It **is** a guarantee that the corpus shows the playbook is used across many domains, motivations, and eras.
- It **is** a guarantee that no single source or leaning dominates the citations.
- It **is not** a guarantee that domestic political examples are 50/50 left/right. If documented disinformation operations skew asymmetrically in a given period, the corpus reflects that. The intellectually honest move is to *name and date* the asymmetry rather than pad it.
- It **is not** a license to skip cross-spectrum cases when they exist. The corpus should include left-of-center examples (e.g., anti-pharma narratives, certain anti-corporate conspiracy threads, RussiaGate-adjacent overcalls) and right-of-center examples (election denial, anti-vax overlap with libertarian-leaning media). They go in. They just aren't forced into parity.

### 7.5 Voice — what to avoid

| Avoid                           | Use instead                                          |
|---------------------------------|-------------------------------------------------------|
| "Shockingly"                    | (delete)                                              |
| "Debunked"                      | "Retracted [year, source]" / "Contradicted by [study]"|
| "Far-right"/"Far-left" as primary description | Name the group/event, let readers tag        |
| "Sources say"                   | Name the source                                       |
| "It is widely believed"         | Cite who and how widely                               |
| Loaded modifiers ("notorious")  | Strip                                                 |

### 7.6 Entry template

```markdown
---
id: <kebab-case-id>
title: "<plain title — neutral phrasing>"
era: <earliest-era-unlocked>
unlockTriggers:
  - <triggers>
techniques: [<DEPICT ids>]
politicalFlavor: <flavor-tag>
sources:
  - title: "..."
    publisher: "..."
    year: <int>
related: [<other-codex-ids>]
estimatedReadTime: <seconds>
---

Lead paragraph: the central claim or event, dated and located.

Body paragraph(s): mechanism, scale, key actors. Numbers wherever available.

Closing paragraph: what the case demonstrates about the DEPICT technique(s) it's tagged with. Plain language. No moral.
```

## 8. Integration with the Mebro reveal

The educational content layer **escalates into** the reveal — it does not stop at it.

| Phase                       | Educational surface behavior                                              |
|-----------------------------|----------------------------------------------------------------------------|
| Phases 1–3 (MVP)            | Ticker, tooltips, codex unlocking, post-action notifications all active.  |
| Phases 4–5                  | Codex entries get more substantial (Cambridge Analytica, IRA-scale stuff).|
| Foreshadowing (cure 0.6–0.79) | Ticker starts seeding inoculation-research facts. Mebro is mentioned.   |
| Reveal trigger              | Ticker rotates to reveal-only set. Notifications suspended for 60s.       |
| Post-reveal                 | "Counter-techniques" codex section auto-unlocks. Codex icon pulses.       |
| Debrief                     | Each of the 6 technique cards is a *codex deep-link*. Tap → full entry.   |

The codex is the persistent artifact. After the player finishes the run, the codex is still there — they can keep reading. That's the long-tail inoculation effect.

## 9. Order of work (engineering)

1. **Schemas + build-time tooling first.** `content/` directory, the Vite codex plugin, JSON schemas, build-time political-balance test. ~1 day.
2. **Seed corpus into files.** Drop the seed material from the user's prompt into the structure as-is. ~half a day.
3. **Ticker.** Simplest surface, validates the file format end-to-end. ~half a day.
4. **Tooltip extension.** Layer on existing UpgradeCard/PlatformCard. ~half a day.
5. **Codex.** Most code, most polish. ~2 days.
6. **Notifications + triggers.** Wire to existing game events (purchase, synergy, etc.). ~1 day.
7. **Reveal integration.** Hook the rotation/suspension/auto-unlock logic into REVEAL.md choreography. ~half a day.

Total: ~6 dev-days, layered onto v0.2 (post-MVP). The architecture is designed so that v0.1 ships with a **placeholder ticker** that reads from a tiny file (~10 facts) — the rest builds out without re-shaping the engine.

## 10. Open questions

1. **Codex entries during v0.1 MVP, or strict v0.2?** Tooltips ("Based on:") are cheap and high-impact; ticker is cheap. Recommend shipping tooltips + ticker in v0.1 with ~15 facts and ~10 precedents. Defer the codex panel + notifications + lazy-loading machinery to v0.2.
2. **External-link policy.** Source URLs that escape the SPA — should they all `target="_blank" rel="noopener noreferrer"`? Recommend yes, plus a small `↗` glyph so the user knows.
3. **Web-search expansion pass.** The user instructed: expand corpus to 150–300 entries via web_search in a future pass. That's a content-authoring sprint, not a code task. Recommend: do that AFTER v0.2 ships the codex panel — author against a working reader UI.
4. **Translation / i18n.** Codex entries are the largest text surface in the game. If localization comes later, file format already supports it (slot a `lang` directory under `content/codex/`). Out of scope for v0.1/v0.2.
5. **Accessibility audit.** Codex panel must be screen-reader navigable; tooltips need proper ARIA; ticker must announce updates non-aggressively (polite ARIA live region with very long throttle so it doesn't spam SR users). Allocate a half-day for this in v0.2.
